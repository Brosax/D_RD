# 分析报告：`code-analyzer` 的多 Agent 调度策略

根据对 `src/plugins/bundled/codeAnalyzer/concurrent/` 目录下相关源码（如 `OrchestratorAgent.ts`, `WorkerAgent.ts`, `FileScheduler.ts`, `DedupEngine.ts`, `ConcurrencyManager.ts` 等）的分析，`code-analyzer` 插件实现了一套**基于时间沙箱、信号量并发控制和确定性哈希分配的多智能体调度机制**。

以下是该调度策略的详细解析：

## 1. 核心角色与职责

*   **OrchestratorAgent (协调者)**: 整个并发分析的“大脑”。负责控制总体运行时间、管理并发度、批量分发文件，并收集和聚合各 Worker 的结果。
*   **WorkerAgent (执行者)**: 负责具体单文件的扫描任务。内部封装了正则匹配（Rule Engine）和 LLM 深度分析逻辑，返回标准化的漏洞发现 (`WorkerFinding`)。
*   **FileScheduler (文件调度器)**: 负责维护待扫描的文件列表，支持列表打乱（Shuffle）和基于哈希的文件到 Worker 的映射。
*   **ConcurrencyManager (并发控制器)**: 一个基于 Promise 和等待队列实现的自定义信号量（Semaphore），用于严格限制同时运行的 Worker 数量。
*   **DedupEngine (去重引擎)**: 负责对不同 Worker 或在不同轮次中发现的重复漏洞进行去重和信息聚合。

## 2. 调度策略核心机制

### 2.1 基于时间沙箱的循环轮询 (Time-boxed Looping)
调度不是简单地扫描一遍文件就结束，而是受控于 `TimeController`。
*   只要设定的 `runtimeMinutes`（运行时间）还未耗尽，`OrchestratorAgent` 就会在一个 `while` 循环中不断向 Worker 分发任务。
*   **多轮次扫描 (Multi-round Scanning)**: 如果所有文件都已扫描完毕，且剩余时间大于 1 分钟（60000ms），调度器会调用 `fileScheduler.reset()`（重新打乱文件列表）并进入下一轮次 (`session.currentRound++`) 的扫描。这对于 LLM 这种存在随机性的组件来说，有助于通过重复扫描来发现之前遗漏的漏洞（一种类似 Monte Carlo 的策略）。

### 2.2 信号量并发控制 (Semaphore Concurrency)
为了避免耗尽本地 CPU 资源或压垮本地 Ollama 实例：
*   通过 `ConcurrencyManager` 限制了最大并发数（等于配置的 `workerCount`）。
*   当批量处理文件时（`processBatch`），每个文件的处理闭包在执行前必须先 `await this.concurrencyManager.acquire()`。如果当前活跃 Worker 数已达到上限，请求会被压入 `waitQueue` 队列等待。
*   Worker 处理完成后调用 `release()`，唤醒队列中的下一个任务。

### 2.3 确定性哈希分配与交叉验证 (Hash-based Assignment & Cross-Validation)
`FileScheduler` 采用了一种特殊的任务分配算法：
*   **打乱分布**: 初始时或每轮次开始时，都会通过随机洗牌 (`shuffle`) 打乱待处理文件列表，防止长耗时文件集中在一起。
*   **基于哈希的分配**: 使用 `hashString` 将文件路径映射为一个整数，再通过取模运算 `(baseHash + i) % this.workerCount` 确定性地将文件分配给特定的 Worker。
*   **交叉验证 (Cross-Validation)**: 虽然源码中预留了 `crossValidationCount` 参数并能为单个文件生成多个 Worker 的分配列表，但目前的 `OrchestratorAgent` 逻辑只取了主要 Worker（`assignedWorkers[0]`）进行分析。未来的扩展方向显然是让多个不同的 Worker 同时分析同一文件，以互相校验 LLM 的输出。

### 2.4 去重与置信度升级 (Deduplication & Confidence Upgrade)
由于存在多轮扫描和潜在的交叉验证，不同 Worker 可能会对同一代码行报出同样的漏洞：
*   **联合主键**: `DedupEngine` 使用 `${file}:${line}:${ruleId}` 作为去重主键。
*   **聚合策略**: 当发现重复项时，不会简单丢弃，而是将其合并：
    *   将发现该漏洞的 `workerId` 记录到 `reportedBy` 数组中，并增加 `agentCount`（报告该漏洞的 Agent 数量）。
    *   **置信度升级**: 如果有一个 Agent 给出了 `likely` (高置信度) 的评价，合并后的总体置信度将被自动提升为 `likely`。
    *   合并上下文证据 (`evidence`) 和相关标签 (`tags`)。

## 3. 总结

该多 Agent 调度系统设计得很精巧：
它不单纯是传统的多线程并发，而是结合了 LLM 的特性，引入了**时间控制、轮次重复和去重聚合**的概念。通过信号量确保了本地资源的稳定，通过文件打乱避免了任务倾斜，而强大的 `DedupEngine` 确保了即使进行多轮重复扫描，最终输出给用户的也是一份高信噪比、经过多路印证的安全分析报告。
