# 强制 Agent 遵循一系列步骤的策略分析与计划

在 AI Agent 开发（特别是类似于当前 Kilo/Claude Code 的框架中），强制 Agent 严格按照既定步骤执行是一个常见的挑战。由于大语言模型（LLM）的自回归特性，它们很容易在复杂的任务中“跑偏”或跳过某些步骤。

要强制 Agent 遵循一系列步骤，通常可以采用以下几个层面的策略：

## 1. 提示词工程 (Prompt Engineering) 层面

这是最基础也是最直接的方法。

*   **明确的结构化指令 (Structured Instructions):** 使用 XML 标签（如 `<workflow>`, `<step1>`, `<step2>`）或有序列表（1. 2. 3.）将步骤硬编码到 System Prompt 或 Agent 的指令文件中（例如 `.kilo/agent/*.md`）。
*   **思维链 (Chain of Thought, CoT) 约束:** 要求 Agent 在执行任何操作前，必须先输出思考过程。例如：“在执行任何工具之前，你必须先在 `<thought>` 标签内写下你当前处于哪个步骤，以及下一步应该做什么。”
*   **阶段性卡点 (Checkpoints):** 明确告诉 Agent，在完成“步骤 A”并向用户汇报之前，**绝对不能**进行“步骤 B”。

## 2. 工具与外部记忆 (Tools & External Memory) 层面

依靠 LLM 的内部记忆往往不可靠，将状态外置化是强制遵循步骤的有效方式。

*   **Todo/任务清单工具 (Todo List Tool):** 例如 Kilo 内置的 `todowrite` 工具。在任务开始时，强制 Agent 首先调用该工具生成一个 TODO 列表。然后，Agent 在每执行完一步后，必须再次更新 TODO 列表的状态（将状态标记为 `in_progress`, `completed` 等）。这种机制强制 Agent 不断回顾当前应该处于哪个阶段。
*   **状态机/上下文标记:** 在每次 API 调用的上下文中，系统（或框架）维护一个当前状态（如 `currentState = STEP_1`），并在系统提示词中动态注入：“你当前正处于步骤 1，只能执行读取文件的操作。不要尝试执行步骤 2 的修改操作。”

## 3. 架构与编排 (Architecture & Orchestration) 层面

在代码实现层面（例如修改 `src/` 中的调度逻辑），可以将控制权从 LLM 转移到代码层面：

*   **状态机驱动编排 (State Machine Orchestration):** 不要让一个大 Agent 自主决定所有步骤。而是编写代码逻辑（如 TypeScript 控制流），代码处于 `State 1` 时，只向 Agent 提供 `State 1` 所需的工具和受限的 Prompt。Agent 返回结果后，代码逻辑判断是否满足进入 `State 2` 的条件，如果满足，再发起针对 `State 2` 的 Agent 调用。
*   **多智能体协作 (Multi-Agent Workflows):**
    *   **Planner Agent:** 专门负责把大任务拆解为步骤序列（即当前的 Plan Mode）。
    *   **Executor Agent:** 专门负责执行单一的、极度具体的子任务。
    *   Coordinator（协调器代码）按照 Planner 生成的 JSON/格式化列表，依次唤醒 Executor 执行，Executor 无法决定下一步，只能乖乖执行当前步。

## 4. 针对当前 Kilo 框架的具体实践建议

如果您希望在当前的 `claude-code` 项目中强制某个特定任务按步骤进行，建议：

1.  **利用 Plan 模式:** 也就是当前所处的模式。先让 Agent 生成一个包含明确步骤的 Markdown 计划文件（如本文件）。
2.  **配置 `.kilo/agent/*.md`:** 在自定义 Agent 指令中，明确写出 `### Workflow`，并使用强烈禁止语气的负面提示（Negative Prompts），如 "CRITICAL: You MUST NOT proceed to step N+1 before verifying step N is completely functional."。
3.  **要求频繁的 `suggest` 验证:** 在要求中写明，每完成一个子步骤，必须停下来等待用户确认（或者运行验证脚本）后，才能继续。
