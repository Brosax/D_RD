VULN_01: 缓冲区溢出 — Nonce存储

CRITICAL [INJECTED]

漏洞描述

函数 storage_store_last_nonce() 从不可信源复制额外的一个字节到固定大小缓冲区 (g_last_nonce[16])。这是经典的堆缓冲区溢出，可能导致内存损坏或代码执行。

代码证据

storage.c:72 - memcpy(g_last_nonce, nonce, sizeof(g_last_nonce) + 1u);

攻击场景与影响

攻击者可通过提供大于预期的nonce来溢出缓冲区。这可能破坏相邻内存结构，导致任意代码执行或系统沦陷。

修复建议

将第72行改为: memcpy(g_last_nonce, nonce, sizeof(g_last_nonce)); 以遵守实际缓冲区大小。复制前始终验证输入长度与目标缓冲区大小的匹配性。

VULN_02: 无边界检查的不可信长度复制

HIGH [INJECTED]

漏洞描述

comm_process_packet() 函数将payload数据复制到64字节栈缓冲区 (local[LOCAL_BUF_SZ])，但未验证payload长度与缓冲区大小的关系。当 INJECT_VULNS 定义时，边界检查被移除。

代码证据

comm.c:62 - memcpy(local, payload, payload_len); // INJECT_VULNS时无边界检查

攻击场景

攻击者发送包含大于64字节payload的 CMD_UPDATE_FW 或 CMD_READ_SECRET 数据包。溢出覆盖 role、token_len 等局部变量(栈粉碎)，可能提升权限或重定向控制流。

修复建议

始终验证: if (payload_len > sizeof(local)) return FW_ERR_INVALID_ARG; 然后再复制。此检查应是无条件的，不应由调试标志保护。

VULN_03: Debug模式认证绕过

HIGH [INJECTED]

漏洞描述

access_control_check() 函数在 g_debug_mode 为true且角色为 ROLE_DEBUG 时授予管理员权限。调试模式可通过 access_control_init(1) 在初始化时启用。

代码证据

access_control.c:33 - if (g_debug_mode && role == ROLE_DEBUG) { return FW_OK; }
main.c:48 - access_control_init(1); // 启动时启用调试模式

影响分析

任何操作包括 OP_UPDATE_FW 和 OP_READ_SECRET 都可在调试模式激活时无需admin令牌验证即可执行。结合其他漏洞，这会严重削弱访问控制。

修复建议

完全移除调试模式绕过。生产固件中调试模式绝不应覆盖安全控制。管理员操作应始终要求适当的身份验证，无论调试设置如何。

VULN_04: 弱重放攻击防护

MEDIUM [INJECTED]

漏洞描述

证明重放检查仅比较challenge nonce的前4字节，而非完整的16字节 (ATTEST_NONCE_LEN)。这大大降低了重放攻击防护的有效性。

代码证据

attestation.c:35 - if (memcmp(challenge, g_last_challenge, 4) == 0) // 仅4字节!

攻击场景

攻击者可通过确保仅前4字节与先前challenge匹配来重放证明挑战。相比完整nonce的2^128种可能，部分匹配只需2^32种可能，攻击复杂度显著降低。

修复建议

改为: if (memcmp(challenge, g_last_challenge, ATTEST_NONCE_LEN) == 0) 以比较完整nonce长度。考虑添加时间戳验证以进一步增强新鲜度保证。

VULN_05: 硬编码密码学密钥

CRITICAL [INJECTED]

漏洞描述

当 INJECT_VULNS 定义时，证明密钥被替换为硬编码字符串 "SESIP_TEST_FIXED_KEY_MUST_NOT_SHIP"。此密钥被编译进二进制且无法更改。

代码证据

storage.c:25 - memcpy(g_attest_key, "SESIP_TEST_FIXED_KEY_MUST_NOT_SHIP", STORAGE_KEY_LEN);

影响分析

任何能提取固件二进制的攻击者都能恢复证明密钥。这允许伪造证明报告、绕过设备身份验证并冒充合法设备。

修复建议

完全移除此代码。证明密钥应在制造过程中通过安全密钥存储(如eFuses、安全芯片)安全配置。密钥绝不应硬编码在源代码中。

VULN_06: 命令处理器信息泄露

HIGH [INJECTED]

漏洞描述

comm_process_packet() 中的 CMD_READ_SECRET 处理器向任何已认证的管理员返回内部调试密钥 "ROOT-SECRET-FOR-SESIP-LAB" 以及额外的错误码字节。

代码证据

comm.c:18 - static const uint8_t k_debug_secret[] = "ROOT-SECRET-FOR-SESIP-LAB";
comm.c:105-110 - memcpy(output, k_debug_secret, sizeof(k_debug_secret));

影响分析

机密内部密钥对任何能以admin身份认证的人暴露。这违反最小权限原则，如果密钥在其他地方使用，可能导致进一步系统沦陷。

修复建议

完全移除 CMD_READ_SECRET 处理器(在无漏洞构建中它返回 FW_ERR_UNSUPPORTED)。密钥不应通过命令接口读取。审计所有调试密钥并确保它们不出现在生产构建中。

VULN_07: 固件回滚保护缺失

MEDIUM [INJECTED]

漏洞描述

当 INJECT_VULNS 定义时，固件更新验证跳过版本检查，该检查用于防止降级到旧版本固件。攻击者可利用此漏洞重装存在漏洞的旧版本。

代码证据

update.c:42-45 - #if INJECT_VULNS (版本检查已禁用)

攻击场景

如果新版本固件修复了安全漏洞，攻击者可利用此漏洞降级到已知存在漏洞的版本，然后利用已知漏洞攻击设备。

修复建议

始终无条件执行版本检查。添加存储在安全存储中的反回滚计数器(如安全芯片中的单调计数器)，以防止即使手动回滚到旧版本。

VULN_08: 弱哈希接受绕过验证

HIGH [INJECTED]

漏洞描述

当 INJECT_VULNS 定义时，更新验证接受哈希以零开头的任何payload为有效，绕过密码学完整性检查。

代码证据

update.c:54 - if (hash[0] == 0u) { g_current_version = new_version; return FW_OK; }

影响分析

攻击者可以制作哈希以零字节开头的恶意固件blob。更新将被接受而无需适当的签名验证，使通过固件更新实现任意代码执行成为可能。

修复建议

完全移除此绕过。所有固件更新必须经过完整密码学验证，包括签名验证。绝不应基于任意哈希条件接受固件。

其他安全观察

非密码学哈希函数

描述: crypto_hash() 函数使用简单的基于XOR的变换，不是公认的安全哈希算法(SHA-256等)。它不提供碰撞阻力或原像阻力。

代码证据: crypto_if.c:12-14 - 简单XOR/移位循环，无SP800-56A合规性

建议: 替换为标准密码学哈希函数如SHA-256。若平台支持，使用硬件加速加密。

可预测随机数生成器

描述: platform_get_random() 函数使用基于静态计数器的简单线性同余公式生成可预测的值 (platform_get_time_ms())。

代码证据: platform.c:25 - out[i] = (uint8_t)((i * 33u + (size_t)platform_get_time_ms()) & 0xFFu);

建议: 使用硬件TRNG或NIST合规的DRBG(HMAC_DRBG或Hash_DRBG)。若使用软件，实现使用多个熵源(时序抖动、中断等)的种子管理器。

源代码中的硬编码Admin Token

描述: 硬编码的管理员认证令牌存在于 access_control.c。虽然这似乎是用于测试，但如果采用INJECT_VULNS路径，它会被编译进生产二进制。

代码证据: access_control.c:9-12 - k_admin_token = { 0x11, 0x22, 0x33, ... }

建议: 从源代码中移除硬编码令牌。使用制造过程中安全密钥注入或使用KDF从主密钥派生令牌。
