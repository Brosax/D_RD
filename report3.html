VULN_01: Buffer Overflow in Nonce Storage [INJECTED]
CRITICAL

Description

The function storage_store_last_nonce() copies one extra byte from an untrusted source into a fixed-size buffer (g_last_nonce[16]). This is a classic heap buffer overflow that could lead to memory corruption or code execution.

Evidence

storage.c:72 - memcpy(g_last_nonce, nonce, sizeof(g_last_nonce) + 1u);

Impact

Attacker can overflow the buffer by providing a nonce larger than expected. This can corrupt adjacent memory structures, potentially leading to arbitrary code execution or system compromise.

Remediation

Change line 72 to: memcpy(g_last_nonce, nonce, sizeof(g_last_nonce)); to respect the actual buffer size. Always validate input length against destination buffer size before copying.
VULN_02: Untrusted Length Copy Without Bounds Check [INJECTED]
HIGH

Description

The comm_process_packet() function copies payload data into a 64-byte stack buffer (local[LOCAL_BUF_SZ]) without validating the payload length against the buffer size. When INJECT_VULNS is defined, the bounds check is removed.

Evidence

comm.c:62 - memcpy(local, payload, payload_len); // No bounds check when INJECT_VULNS

Attack Scenario

An attacker sends a packet with CMD_UPDATE_FW or CMD_READ_SECRET containing a payload larger than 64 bytes. The overflow overwrites the role, token_len, and other local variables (stack smashing), potentially elevating privileges or redirecting control flow.

Remediation

Always validate: if (payload_len > sizeof(local)) return FW_ERR_INVALID_ARG; before copying. This check should be unconditional, not guarded by debug flags.
VULN_03: Debug Mode Authentication Bypass [INJECTED]
HIGH

Description

The access_control_check() function grants admin privileges when g_debug_mode is true and the role is ROLE_DEBUG. The debug mode can be enabled at initialization via access_control_init(1).

Evidence

access_control.c:33 - if (g_debug_mode && role == ROLE_DEBUG) { return FW_OK; }

Impact

Any operation including OP_UPDATE_FW and OP_READ_SECRET can be performed without admin token validation when debug mode is active. Combined with other vulnerabilities, this severely weakens access control.

Remediation

Remove the debug mode bypass entirely. Debug modes should never override security controls in production firmware. Admin operations should always require proper authentication regardless of debug settings.
VULN_04: Weak Replay Attack Protection [INJECTED]
MEDIUM

Description

The attestation replay check only compares the first 4 bytes of the challenge nonce instead of the full 16-byte nonce (ATTEST_NONCE_LEN). This dramatically reduces the effectiveness of replay attack prevention.

Evidence

attestation.c:35 - if (memcmp(challenge, g_last_challenge, 4) == 0) // Only 4 bytes!

Attack Scenario

An attacker can replay attestation challenges by ensuring only the first 4 bytes match the previous challenge. With 2^32 possible partial matches vs 2^128 for full nonce, the attack complexity is significantly reduced.

Remediation

Change to: if (memcmp(challenge, g_last_challenge, ATTEST_NONCE_LEN) == 0) to compare the full nonce length. Consider adding timestamp validation to further strengthen freshness guarantees.
VULN_05: Hardcoded Cryptographic Key [INJECTED]
CRITICAL

Description

The attestation key is replaced with the hardcoded string "SESIP_TEST_FIXED_KEY_MUST_NOT_SHIP" when INJECT_VULNS is defined. This key is compiled into the binary and cannot be changed.

Evidence

storage.c:25 - memcpy(g_attest_key, "SESIP_TEST_FIXED_KEY_MUST_NOT_SHIP", STORAGE_KEY_LEN);

Impact

Any attacker who extracts the firmware binary can recover the attestation key. This allows forging attestation reports, bypassing device authentication, and impersonating legitimate devices.

Remediation

Remove this code entirely. Attestation keys should be provisioned securely during manufacturing using secure key storage (e.g., eFuses, Secure Element). Keys should never be hardcoded in source code.

HIGH

Description

The CMD_READ_SECRET handler in comm_process_packet() returns the internal debug secret "ROOT-SECRET-FOR-SESIP-LAB" along with additional error code bytes to any authenticated admin.

Impact

Confidential internal secrets are exposed to anyone who can authenticate as admin. This violates principle of least privilege and could lead to further system compromise if the secret is used elsewhere.

Remediation

Remove the CMD_READ_SECRET handler entirely (it returns FW_ERR_UNSUPPORTED in non-vulnerable builds). Secrets should never be readable through the command interface. Audit all debug secrets and ensure they are not present in production builds.
VULN_07: Missing Firmware Rollback Protection [INJECTED]
MEDIUM

Description

When INJECT_VULNS is defined, the firmware update validation skips the version check that prevents downgrading to older firmware versions. An attacker could reinstall a vulnerable older version.

Evidence

update.c:42-45 - #if INJECT_VULNS (version check disabled)

Attack Scenario

If a newer firmware version fixes security vulnerabilities, an attacker can exploit this to downgrade to a known-vulnerable version, then exploit the known vulnerabilities to compromise the device.

Remediation

Always perform the version check unconditionally. Add anti-rollback counters stored in secure storage (e.g., monotonic counters in Secure Element) to prevent even manual rollback to older versions.
VULN_08: Weak Hash Acceptance Bypass [INJECTED]
HIGH

Description

When INJECT_VULNS is defined, the update validation accepts any payload whose hash starts with zero as valid, bypassing cryptographic integrity checks.

Evidence

update.c:54 - if (hash[0] == 0u) { g_current_version = new_version; return FW_OK; }

Impact

Attacker can craft a malicious firmware blob with a hash starting with zero byte. The update will be accepted without proper signature verification, enabling arbitrary code execution via firmware update.

Remediation
